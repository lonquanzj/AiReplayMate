# AiReplayMate 代码评审

在评审 AiReplayMate 代码改动时使用本技能。

## 目标

输出项目特定评审，而不是通用 Android 评审。所有结论应以当前 MVP 范围和仓库既有工程约束为依据。

## 必要上下文

开始评审前，先加载最小必要上下文：

1. 阅读 [docs/CODE_REVIEW.md](../../../docs/CODE_REVIEW.md)，作为评审基线。
2. 若改动可能影响产品范围或 MVP 边界，阅读 [docs/PRD.md](../../../docs/PRD.md)。
3. 若改动涉及架构、适配器、Accessibility、OCR、LLM、自动填入、存储或日志，阅读 [docs/ENGINEERING_SPEC.md](../../../docs/ENGINEERING_SPEC.md)。
4. 若改动可能默认依赖尚未落地的模块或基础设施，阅读 [docs/CURRENT_STATUS.md](../../../docs/CURRENT_STATUS.md)。

只收集与本次改动切片相关的上下文，不要在评审前做全仓库扫描。

## 评审优先级

优先检查高风险问题：

1. 违反 MVP 边界
2. WeChat 特有逻辑泄漏到适配器边界之外
3. Accessibility、OCR、LLM 或自动填入在无降级时形成单点依赖
4. 任何可能自动发送而非仅自动填入的路径
5. 隐私泄漏、过度日志或不必要持久化聊天数据、截图、API 密钥
6. 缺少失败处理、状态收口或诊断信号
7. 本次触达行为的验证缺口

## 项目特定检查问题

用以下问题驱动评审：

- 改动是否仍符合当前 MVP：微信单聊、辅助回复、仅自动填入、不自动发送？
- 逻辑是否放在正确分层，还是耦合进了 Activity、Compose UI 或 AccessibilityService？
- 微信识别与解析规则是否隔离在适配器抽象之后？
- 流程是否仍遵循 Accessibility 优先、OCR 仅兜底？
- 自动填入是否采用安全机制，并规避发送动作或等效隐式发送？
- 失败是否明确、可恢复、可观测？
- 本次改动是否导致仓库文档描述失真？

## 输出格式

输出评审结果时：

1. 先列发现项，并按严重级别排序。
2. 聚焦缺陷、回归风险、设计约束违背与验证缺失。
3. 尽量提供具体文件定位。
4. 总结应简短并置于次要位置。
5. 若无发现项，要明确说明，并补充剩余测试或发布风险。

## 评审结论指引

- 出现以下情况应阻断合并：MVP 边界破坏、适配器边界破坏、自动发送风险、关键流程缺少降级路径、隐私或安全问题。
- 出现以下情况应要求修改：状态收口不足、诊断能力薄弱、风险路径验证不充分。
- 仅当改动与当前范围一致、遵守架构约束且验证强度与风险匹配时才建议通过。
