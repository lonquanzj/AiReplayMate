# 真机关键场景补齐计划 — AiReplayMate

> 本文档回答两个问题：在当前已经具备稳定无 Activity 回归基线的前提下，哪些真机关键场景仍未补齐？后续应该按什么顺序逐步补齐？
>
> 目标不是替代现有稳定 smoke，而是在其之上补齐更接近真实使用的真机链路验证。

## 1. 背景

当前仓库已经具备一条稳定的“无 Activity 真机回归”基线：

- 默认 smoke 不依赖 `MainActivity` 启动
- 已覆盖 Prompt、Local Fallback、LLM Debug、Overlay Debug、DiagnosticLog、OCR Debug 等核心纯逻辑 / 状态流
- 可在当前 MIUI 真机上稳定执行，不容易被系统后台启动限制干扰

但这条基线并不等于真实使用链路。

真实用户使用时，仍然依赖以下真机条件：

1. 无障碍权限真实可用
2. 悬浮窗权限真实可用
3. 无障碍截图能力真实可用
4. 微信页面识别与消息提取在当前微信版本下真实可用
5. OCR 截图与识别在当前手机主题 / 字体 / 分辨率下真实可用
6. LLM 真实连通与候选返回真实可用
7. Autofill / 输入框填入在当前 ROM 上真实可用

因此需要补一套“真机关键场景计划”，把稳定回归与真实链路验证明确分层。

## 2. 分层策略

### 层 1：稳定基线回归

用途：保证代码层、规则层、状态层不回退。

当前入口：

- `scripts/run-device-smoke-test.ps1`
- VS Code 任务：`device smoke: stable suite`

这层默认不追求“像真实用户一样点开 App 并走完微信全链路”，而是追求：

- 稳定
- 可重复
- 不容易被 MIUI / 系统策略噪声打断

### 层 2：真实链路专项验证

用途：验证当前真机上的真实使用体验。

这层允许不稳定，因为它更接近真实系统环境。

包括：

- 权限联动
- 悬浮服务生命周期
- 微信页面识别
- OCR 真机兜底
- 真实 LLM 连通
- 候选展示与真实填入

执行入口（Diagnostic Only，不作为 merge gate）：

- VS Code 任务：`device diagnostic: ui entry probe (diagnostic only)`
- VS Code 任务：`device diagnostic: activity launch probe (diagnostic only)`
- 脚本命令：`scripts/run-device-smoke-test.ps1 -TestClass com.lonquanzj.aireplaymate.MainActivityUiEntryTest#mainScreen_renders_key_entry_points`
- 脚本命令：`scripts/run-device-smoke-test.ps1 -TestClass com.lonquanzj.aireplaymate.MainActivityLaunchProbeTest#mainActivity_launches_with_activityScenario`

推荐执行顺序：

1. 先跑 UI entry probe，确认阶段化入口日志
2. 再跑 activity launch probe，确认是否命中 MIUI / 系统后台启动限制

### 层 3：人工冒烟

用途：在目标自用手机上确认“今天还能不能顺手使用”。

这层不追求自动化完备，而追求：

- 快速确认
- 可解释失败
- 能指导下一步排障

## 3. 补齐原则

后续补场景时，遵循以下原则：

1. 先补前置条件，再补端到端链路
2. 先补诊断可见性，再补自动化程度
3. Activity 启动类用例继续保留为专项诊断，不回到默认 smoke 主入口
4. 所有真实链路场景都要附带失败分类，而不只写“通过 / 失败”

## 4. 开发阶段

### 阶段 0：冻结稳定基线

目标：确保当前无 Activity 稳定 smoke 继续作为回归底座。

工作项：

1. 保持默认稳定 smoke 为当前真机基础回归入口
2. 避免把易受 MIUI 限制的 Activity 启动测试重新放回默认 smoke
3. 所有新增真实链路验证都以“不影响稳定基线”为前提

完成标准：

1. 默认 smoke 持续可在当前真机跑绿
2. 后续新增真机场景不会污染当前稳定回归入口

### 阶段 1：前置条件与权限联动

目标：先确认真实链路前提是否满足。

工作项：

1. 验证无障碍未开启时首页提示和诊断是否正确
2. 验证悬浮窗未开启时首页提示和服务状态是否正确
3. 验证无障碍截图不可用时的诊断路径是否清楚
4. 验证服务被系统回收时首页是否能体现失效状态

完成标准：

1. 能快速区分“权限没开”和“代码逻辑错误”
2. 首页诊断对前置条件失败给出可操作反馈

### 阶段 2：悬浮服务真实生命周期

目标：验证 OverlayButtonService 在当前 MIUI 真机上的可用性与恢复性。

工作项：

1. 从首页启动悬浮服务
2. 从首页停止悬浮服务
3. 切到微信后确认气泡仍可见
4. 锁屏 / 熄屏 / 切后台后确认是否被系统回收
5. 被回收后确认能否快速恢复或至少被诊断出来

完成标准：

1. 用户能稳定看到气泡
2. 服务失效时能通过首页诊断快速定位问题

### 阶段 3：微信页面识别与上下文提取

目标：确认当前微信版本下，真实单聊页识别与 Accessibility 上下文提取可用。

工作项：

1. 补标准微信单聊页样本
2. 验证非聊天页误触发时是否被拦截
3. 验证 Accessibility 可用时的消息提取
4. 验证消息不足时能否正确进入 OCR 兜底判断

完成标准：

1. 单聊页识别稳定
2. 非单聊页不误触发主链路
3. 上下文不足与提取失败可区分

### 阶段 4：OCR 真机兜底链路

目标：从状态验证升级到真实截图 + OCR + 后处理链路验证。

工作项：

1. Accessibility 不足时进入 OCR
2. 验证无障碍截图成功时可识别
3. 验证 OCR 无文本时诊断是否合理
4. 验证 OCR 输出能进入上下文合并

完成标准：

1. OCR 在当前手机上至少具备可用兜底能力
2. 失败时能明确区分是截图、识别还是后处理问题

### 阶段 5：真实 LLM 连通与候选生成

目标：验证真实网络、真实模型、真实候选返回，而不只是 DebugStore 状态。

工作项：

1. 测试连接成功路径
2. Base URL 错误路径
3. API Key 错误路径
4. Model 不存在路径
5. 真实单聊上下文下生成候选
6. LLM 失败后真实切换本地兜底

完成标准：

1. 用户能明确区分配置错误、网络错误、服务错误
2. 候选能真实返回并进入后续展示链路

### 阶段 6：候选展示与真实填入

目标：补齐最接近用户感知的结果层。

工作项：

1. 候选面板展示是否正常
2. 候选来源标记是否正确
3. 用户选择候选后输入框填入是否成功
4. 填入失败时是否给出合理诊断
5. 是否存在复制或其他兜底路径

完成标准：

1. 候选能稳定写入输入框
2. 失败不会变成无反馈黑盒

### 阶段 7：真实链路专项诊断

目标：保留受 MIUI / 系统策略影响的链路，但不把它们作为默认 smoke。

工作项：

1. 保留 `MainActivity` 启动探针
2. 保留 MIUI 后台启动限制诊断
3. 明确区分“代码回归”和“系统策略拦截”

完成标准：

1. Activity 启动失败时能快速确认是否是 MIUI / 系统限制
2. 不再把这类诊断结果误判为主链路功能损坏

### 阶段 8：文档与任务收敛

目标：让后续补场景的人能快速上手并按分层执行。

工作项：

1. 维护本计划文档
2. 在 README 中继续区分稳定 smoke 与真实链路验证
3. 为常用真实链路验证补 VS Code 任务入口
4. 形成一套当前自用真机的手工冒烟步骤

完成标准：

1. 后续推进时可以直接按阶段执行
2. 失败时能快速判断属于哪一层问题

## 5. 优先级建议

按实际价值排序，建议优先补齐顺序如下：

1. 前置条件与权限联动
2. 悬浮服务真实生命周期
3. 微信页面识别与上下文提取
4. OCR 真机兜底链路
5. 真实 LLM 连通与候选生成
6. 候选展示与真实填入
7. 真实链路专项诊断
8. 文档与任务收敛

## 6. 每阶段固定产出

为了避免后续计划执行中再次散掉，建议每推进一个阶段，都固定产出：

1. 场景清单
2. 成功标准
3. 失败分类
4. 当前结论

## 7. 当前建议

如果下一步继续推进，我建议先做：

1. 阶段 1：前置条件与权限联动
2. 阶段 2：悬浮服务真实生命周期

这是因为当前最容易影响日常可用性的，并不是 Prompt 或 DebugStore，而是：

- 服务是否还活着
- 权限是否满足
- 失败时能不能立刻定位原因
