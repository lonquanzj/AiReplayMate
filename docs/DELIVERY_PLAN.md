# 交付计划 — AiReplayMate

> 本文档合并原 `ROADMAP.md` 与 `TASK_BREAKDOWN.md`，统一维护版本路线、任务拆解、依赖关系与粗排期。产品边界以 `PRD.md` 为准，工程方案与实现约束以 `ENGINEERING_SPEC.md` 为准。
>
> 当前中期定位：一台手机上的个人自用单人项目。近期优先级应服务“自用稳定、少维护、能排障”，不按多机型发布、团队协作或商业产品化来排期。

## 1. 版本路线

### V1.0 自用 MVP（目标：先稳定个人主链路）

> 核心链路跑通：触发 → 提取 → 生成 → 填入

- [ ] 项目基础架构保持轻量，暂不强求 Gradle 多模块、DI、复杂导航
- [x] 权限引导与状态诊断（悬浮窗、无障碍）
- [x] 悬浮触发按钮（FloatingButton）
- [x] AccessibilityService 骨架 + 微信聊天页识别
- [x] Accessibility 消息提取（微信单聊）
- [x] 无障碍截图 + ML Kit OCR 兜底
- [x] ContextBuilder 合并上下文
- [x] LLM API 对接（OpenAI 兼容接口）
- [x] 候选回复展示面板（3 条）
- [x] 自动填入输入框（`ACTION_SET_TEXT` + 剪贴板兜底）
- [x] 设置页面（API Key、模型选择、Base URL）
- [x] 基本日志与诊断面板
- [ ] 端到端自测（当前自用手机 + 当前微信版本）

**V1.0 完成标志**：在开发者手机上可稳定完成一次完整的“触发 → 提取 → 生成 → 填入”流程。

当前落地状态：

- V1.0 主链路已在当前自用手机上跑通，最近一次稳定 smoke 为 7 条用例通过。
- 2026-05-23 再次回归：稳定 smoke 7/7 通过，`:app:testDebugUnitTest` 通过。
- 同期两个 Diagnostic Only 的 Activity 探针在当前 MIUI 设备失败（后台拉起 Activity 限制），不作为 merge gate。
- OCR 兜底已从 MediaProjection 切换为无障碍截图，避免频繁截图授权和后续帧模糊问题。
- OCR 诊断已能展示截图来源、帧质量、过滤摘要、聚合消息和 Debug 调试图路径。
- OCR 调试图仅 Debug 构建保存有限数量，Release / 非调试构建不保存。
- LLM Prompt 仅在实际发送 OCR 上下文时追加 OCR 纠错提示，纯 Accessibility 上下文不追加。

### V1.1 自用体验增强

- [ ] 候选回复二次调整（缩短 / 更礼貌 / 更正式）
- [ ] 轻量候选历史记录（优先服务排障和复用，不追求完整会话管理）
- [ ] 常用联系人或常用场景的回复偏好
- [ ] 回复内容长度控制（用户可设偏好长短）
- [ ] 当前手机上的性能优化（OCR 速度、启动速度、悬浮窗恢复）

### V1.2 可选扩展

- [ ] 基础群聊支持（`@` 提及检测、上下文截取策略）
- [ ] 屏幕区域选择（用户手动框选 OCR 区域）
- [ ] 更多聊天 App 适配器（企业微信、钉钉、短信）
- [ ] 更强的隐私脱敏（自动识别并遮蔽手机号/身份证/银行卡）
- [ ] 国际化：英文界面 + 多语言回复支持

### V1.3 表情包 / 图片回应能力

> 目标：先把“对方发了表情/图片”纳入上下文，再逐步升级到视觉理解与本地表情库推荐。默认保持用户确认后发送，不进入自动发送主链路。

- [ ] 第一版：不用 Vision，只识别“对方发了表情 / 图片”，作为结构化占位消息进入上下文
- [ ] 提取对方最新图片 / 表情消息的屏幕区域，只裁剪必要区域，不上传整屏截图
- [ ] 接入可选的阿里云百炼 `qwen3-vl-flash` 视觉分类，用于判断表情包 / 图片的大致语气意图
- [ ] 建立 `VisualMessageIntent`：开心、暧昧、委屈、生气、疑问、无语、可爱、未知等粗分类
- [ ] 建立本地表情库：支持 emoji、微信文本表情、本地图片 / 动图表情包，并维护标签、强度、场景
- [ ] 候选面板支持展示文字回复、emoji、微信文本表情和本地图片表情包推荐
- [ ] 支持在软件内上传 / 导入表情包，给表情包打标签并参与后续推荐
- [ ] 探索图片表情发送能力：优先用户确认 + 手动发送 / 分享面板兜底，再评估更自动化方案

### V2.0 长期想法

- [ ] 本地小模型支持（通过 ML Kit 或 `onnxruntime` 运行量化模型）
- [ ] 个性化语气学习（分析用户历史回复风格）
- [ ] 按联系人独立配置（不同联系人不同回复风格）
- [ ] 多轮上下文优化（更长对话历史 + 关键信息提取）
- [ ] 快捷短语 / 常用回复模板库
- [ ] 可选的自动发送模式（设置中手动开启，风险提示）

## 2. MVP 任务拆解

> 优先级定义：P0 = MVP 必须完成；P1 = 核心但可滞后；P2 = 体验完善。

### P0 — MVP 骨架（基础链路）

| # | 任务 | 依赖 | 预估 | 说明 |
|---|------|------|------|------|
| 1 | 项目初始化：保持单模块轻量结构 | 无 | 0.5d | 单人项目优先少维护；多模块、Hilt、复杂导航延后 |
| 2 | 权限引导页面（悬浮窗、无障碍） | 无 | 1d | 引导用户开启必要权限，图文说明 |
| 3 | 悬浮按钮 FloatingButton（Overlay） | #2 | 1d | 系统级悬浮窗，常驻显示，可拖动 |
| 4 | AccessibilityService 骨架 + 微信页面识别 | #2 | 2d | 注册 Service，通过包名 + 节点结构判读页面 |
| 5 | 候选回复展示面板（BottomSheet） | 无 | 2d | 3 条候选横向展示，点击选择 / 重新生成 |
| 6 | AutofillEngine（`ACTION_SET_TEXT` + 剪贴板兜底） | #4 | 1.5d | 自动填入 + 验证 + 降级 |
| 7 | SessionManager 主流程状态机 | #3 #4 #5 #6 | 2d | 串联触发 → 收集 → 处理 → 展示 → 填入全流程 |

**P0 总计：约 10 人天**

### P1 — 核心能力（提取 + LLM）

| # | 任务 | 依赖 | 预估 | 说明 |
|---|------|------|------|------|
| 8 | Accessibility 消息提取器（微信单聊） | #4 | 3d | 遍历消息节点、解析内容 / 时间 / 发送方 |
| 9 | 输入框状态读取（是否有文本、是否可输入） | #4 | 0.5d | 用于自动填入前后验证 |
| 10 | 无障碍截图流程 | #2 #4 | 1.5d | 通过 `AccessibilityService.takeScreenshot` 截图，避免频繁授权 |

| 12 | OCR 后处理（按坐标排序、区分左右气泡） | #11 | 1.5d | 将 OCR 结果组装为结构化消息列表 |
| 13 | ContextBuilder（合并去重、截取策略） | #8 #12 | 1d | 统一 Accessibility + OCR 输出 |

## 4. 排期建议

- 并行开发时，P0 的 `#5 候选面板` 可与 `#3 + #4` 同步推进。
- P1 的依赖链较长，建议优先启动 `#8 ~ #12`，尽快打通“上下文采集”。
- 单人自用时，建议先做 `P0 + P1 的 #8 #13 #14 #15`，把最短可用闭环跑起来，再围绕当前手机补 OCR、Autofill、悬浮窗生命周期和诊断能力。
- 表情包 / 图片回应不要阻塞 MVP 主链路；建议先做 `#24` 占位识别，确认上下文和候选体验顺手后，再接 `#26` 百炼视觉分类，最后推进 `#27~#29` 本地表情库与发送能力。
- 不急于推进多模块、Hilt、Room、跨厂商兼容矩阵；只有当单模块维护成本明显上升、或个人使用确实需要历史数据时再引入。
