# 交付计划 — AiReplayMate

> 本文档合并原 `ROADMAP.md` 与 `TASK_BREAKDOWN.md`，统一维护版本路线、任务拆解、依赖关系与粗排期。产品边界以 `PRD.md` 为准，工程方案与实现约束以 `ENGINEERING_SPEC.md` 为准。
>
> 当前中期定位：一台手机上的个人自用单人项目。近期优先级应服务“自用稳定、少维护、能排障”，不按多机型发布、团队协作或商业产品化来排期。

## 1. 版本路线

### V1.0 自用 MVP（目标：先稳定个人主链路）

> 核心链路跑通：触发 → 提取 → 生成 → 填入

- [ ] 项目基础架构保持轻量，暂不强求 Gradle 多模块、DI、复杂导航
- [ ] 权限引导页面（悬浮窗、无障碍、截图）
- [ ] 悬浮触发按钮（FloatingButton）
- [ ] AccessibilityService 骨架 + 微信聊天页识别
- [ ] Accessibility 消息提取（微信单聊）
- [ ] 截图 + ML Kit OCR 兜底
- [ ] ContextBuilder 合并上下文
- [ ] LLM API 对接（OpenAI 兼容接口）
- [ ] 候选回复展示面板（3 条）
- [ ] 自动填入输入框（`ACTION_SET_TEXT` + 剪贴板兜底）
- [ ] 设置页面（API Key、模型选择、Base URL）
- [ ] 基本日志与诊断面板
- [ ] 端到端自测（当前自用手机 + 当前微信版本）

**V1.0 完成标志**：在开发者手机上可稳定完成一次完整的“触发 → 提取 → 生成 → 填入”流程。

### V1.1 自用体验增强

- [ ] 候选回复二次调整（缩短 / 更礼貌 / 更正式）
- [ ] 轻量候选历史记录（优先服务排障和复用，不追求完整会话管理）
- [ ] 常用联系人或常用场景的回复偏好
- [ ] 回复内容长度控制（用户可设偏好长短）
- [ ] 当前手机上的性能优化（OCR 速度、启动速度、悬浮窗恢复）

### V1.2 可选扩展

- [ ] 基础群聊支持（`@` 提及检测、上下文截取策略）
- [ ] 屏幕区域选择（用户手动框选 OCR 区域）
- [ ] 更多聊天 App 适配器（企业微信、钉钉、短信）
- [ ] 更强的隐私脱敏（自动识别并遮蔽手机号/身份证/银行卡）
- [ ] 国际化：英文界面 + 多语言回复支持

### V2.0 长期想法

- [ ] 本地小模型支持（通过 ML Kit 或 `onnxruntime` 运行量化模型）
- [ ] 个性化语气学习（分析用户历史回复风格）
- [ ] 按联系人独立配置（不同联系人不同回复风格）
- [ ] 多轮上下文优化（更长对话历史 + 关键信息提取）
- [ ] 快捷短语 / 常用回复模板库
- [ ] 可选的自动发送模式（设置中手动开启，风险提示）

## 2. MVP 任务拆解

> 优先级定义：P0 = MVP 必须完成；P1 = 核心但可滞后；P2 = 体验完善。

### P0 — MVP 骨架（基础链路）

| # | 任务 | 依赖 | 预估 | 说明 |
|---|------|------|------|------|
| 1 | 项目初始化：保持单模块轻量结构 | 无 | 0.5d | 单人项目优先少维护；多模块、Hilt、复杂导航延后 |
| 2 | 权限引导页面（悬浮窗、无障碍、截图三项权限） | 无 | 1d | 引导用户开启必要权限，图文说明 |
| 3 | 悬浮按钮 FloatingButton（Overlay） | #2 | 1d | 系统级悬浮窗，常驻显示，可拖动 |
| 4 | AccessibilityService 骨架 + 微信页面识别 | #2 | 2d | 注册 Service，通过包名 + 节点结构判读页面 |
| 5 | 候选回复展示面板（BottomSheet） | 无 | 2d | 3 条候选横向展示，点击选择 / 重新生成 |
| 6 | AutofillEngine（`ACTION_SET_TEXT` + 剪贴板兜底） | #4 | 1.5d | 自动填入 + 验证 + 降级 |
| 7 | SessionManager 主流程状态机 | #3 #4 #5 #6 | 2d | 串联触发 → 收集 → 处理 → 展示 → 填入全流程 |

**P0 总计：约 10 人天**

### P1 — 核心能力（提取 + LLM）

| # | 任务 | 依赖 | 预估 | 说明 |
|---|------|------|------|------|
| 8 | Accessibility 消息提取器（微信单聊） | #4 | 3d | 遍历消息节点、解析内容 / 时间 / 发送方 |
| 9 | 输入框状态读取（是否有文本、是否可输入） | #4 | 0.5d | 用于自动填入前后验证 |
| 10 | MediaProjection 截图流程 | #2 | 1.5d | 申请 + 截图 + 内存缓存 |
| 11 | ML Kit OCR 引擎集成 | #10 | 2d | 集成 Google ML Kit 文字识别 |
| 12 | OCR 后处理（按坐标排序、区分左右气泡） | #11 | 1.5d | 将 OCR 结果组装为结构化消息列表 |
| 13 | ContextBuilder（合并去重、截取策略） | #8 #12 | 1d | 统一 Accessibility + OCR 输出 |
| 14 | PromptBuilder（组装 LLM Prompt） | #13 | 1d | 含默认 / 自定义 System Prompt |
| 15 | LLM API 集成（OpenAI 兼容） | 无 | 1.5d | OkHttp + Retrofit，超时 / 重试 / 错误处理 |
| 16 | 管道集成联调 | #7 #8~15 | 2d | 全链路打通，修复集成问题 |

**P1 总计：约 14 人天**

### P2 — 体验完善

| # | 任务 | 依赖 | 预估 | 说明 |
|---|------|------|------|------|
| 17 | 设置页面（API Key / Base URL / 模型 / 参数） | 无 | 2d | 偏好存储 + UI 配置页 |
| 18 | 轻量候选历史 / 最近结果 | 本地偏好或轻量存储 | 1d | 个人复用和排障优先，暂不强求 Room |
| 19 | 日志与诊断页面 | 现有 diagnostics | 1d | 展示最近关键阶段、失败原因、复制 / 清除 |
| 20 | 错误处理优化（各阶段异常提示、超时处理） | #7 | 1d | Toast / Snackbar 提示 |
| 21 | 当前手机稳定性优化（内存泄漏、ANR、后台限制适配） | #7 | 1.5d | 优先修复自用手机上真实遇到的问题 |
| 22 | 当前手机回归清单 | 全部 | 1d | 固定当前微信版本、系统版本、主题和权限状态 |
| 23 | 个人隐私检查清单 | 无 | 0.5d | 确认敏感数据不落地、不误传、不自动发送 |

**P2 总计：约 8 人天**

## 3. 依赖关系图

```text
P0:
  #1 项目初始化
  #2 权限引导 ──→ #3 悬浮按钮
  │                #4 AccessibilityService ──→ #6 AutofillEngine
  │                                             │
  #5 候选面板 ◄──────────────────────────────────┘
  │
  └──→ #7 SessionManager（串联 #3 #4 #5 #6）

P1:
  #4 → #8 Accessibility提取器
  #4 → #9 输入框状态读取
  #2 → #10 MediaProjection → #11 ML Kit OCR → #12 OCR后处理
  #8 + #12 → #13 ContextBuilder → #14 PromptBuilder → #15 LLM API
  #7 + #8~15 → #16 集成联调

P2:
  #17 设置页面（独立）
  #18 轻量历史（依赖现有设置或轻量存储）
  #19 日志诊断（依赖现有 diagnostics）
  #20~23 自用稳定性、回归与隐私检查（依赖全链路）
```

## 4. 排期建议

- 并行开发时，P0 的 `#5 候选面板` 可与 `#3 + #4` 同步推进。
- P1 的依赖链较长，建议优先启动 `#8 ~ #12`，尽快打通“上下文采集”。
- 单人自用时，建议先做 `P0 + P1 的 #8 #13 #14 #15`，把最短可用闭环跑起来，再围绕当前手机补 OCR、Autofill、悬浮窗生命周期和诊断能力。
- 不急于推进多模块、Hilt、Room、跨厂商兼容矩阵；只有当单模块维护成本明显上升、或个人使用确实需要历史数据时再引入。
