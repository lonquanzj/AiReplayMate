# Code Review 指南 — AiReplayMate

> 这份文档定义 AiReplayMate 当前阶段的 Code Review 基线。它不是通用 Android 清单，而是围绕本项目的 MVP 目标、工程现状和主要风险制定的评审标准。

## 1. 评审目标

当前仓库仍处于 Android 单模块骨架阶段，评审重点不是“功能做得多不多”，而是：

- 是否沿着 MVP 主链路推进，而不是提前扩散范围
- 是否把高风险能力放在正确的抽象边界里
- 是否为后续接入提取、生成、填入链路保留了可演进空间
- 是否避免引入违反产品边界、安全约束和隐私约束的实现

换句话说，Code Review 的首要目标是保证“方向正确、边界清晰、风险可控”。

## 2. 适用范围

本指南默认适用于以下改动：

- Android 应用代码
- 无障碍服务与页面识别
- 微信适配逻辑
- 自动填入链路
- LLM、OCR、存储、日志等后续能力接入
- 文档中定义的 MVP 主链路相关改动

如果改动与文档定义冲突，以以下顺序判断：

1. PRD.md
2. ENGINEERING_SPEC.md
3. DELIVERY_PLAN.md
4. 本文档

## 3. 评审结论标准

### 3.1 可直接通过

同时满足以下条件时可通过：

- 改动落在当前阶段应做的范围内
- 没有破坏 MVP 边界
- 关键风险有处理或有明确降级
- 接口、状态、错误处理和日志策略基本自洽
- 至少有与改动范围匹配的验证方式

### 3.2 必须拦截

出现以下任一情况，应要求修改后再合入：

- 让产品行为越出 MVP 边界
- 将微信规则散落到通用层，破坏 Adapter 隔离
- 自动发送消息或变相绕过“仅自动填入、不自动发送”约束
- 无降级路径却把 Accessibility、OCR、LLM、Autofill 之一作为单点依赖
- 把敏感聊天内容、API Key、截图等不必要地持久化
- 只堆功能，不补最基本的错误处理、状态收口或可诊断信息
- 文档约束已经变化，但代码没有同步说明

## 4. 项目特定评审清单

### 4.1 产品边界

先确认改动是否仍然符合当前产品定义：

- 是否仍然只面向微信单聊 MVP
- 是否仍然是“辅助回复 + 自动填入”，而不是自动发送
- 是否避免顺手引入群聊、多 App、复杂消息类型等扩范围实现
- 是否没有把未来版本能力硬编码进当前主流程

如果一个 PR 既做当前 P0/P1 任务，又顺带引入 V1.1/V1.2 逻辑，默认应拆分。

### 4.2 架构边界

当前仓库代码仍少，越要严格看边界：

- 新代码是否贴近 ENGINEERING_SPEC.md 里的分层职责
- 当前只是骨架时，是否优先建立接口和状态边界，而不是把实现耦死在 Activity 或 Service 中
- 微信识别与消息提取规则是否收敛在 Adapter，而不是散落在 UI、Service、工具类里
- Autofill、提取、LLM、OCR 是否保持可替换、可降级

特别注意两类反模式：

- 在 ReplyAccessibilityService 中直接堆整条业务链路
- 在 MainActivity 或 Compose 页面里直接写微信解析、权限决策或填入逻辑

### 4.3 状态机与流程收口

本项目主链路天然是状态驱动的，评审要重点看：

- 是否明确区分触发、提取、生成、展示、填入、失败、取消等阶段
- 是否存在超时、失败、用户取消后的收口路径
- 是否避免重复触发、重复展示、重复填入
- 异常发生后，状态是否能回到可再次触发的稳定状态

如果代码里开始出现多个布尔值互相控制流程，这是需要在评审里直接指出的信号。

### 4.4 微信页面识别与适配

这是项目最容易“看起来能跑、实际极脆”的部分：

- 页面识别是否基于包名 + 合理节点特征，而不是单一脆弱条件
- 选择器、类名、文案匹配是否有最小必要性说明
- 是否把微信特有规则封装在 WeChatAdapter 一侧
- 失败时是否明确返回“未识别/不可用”，而不是静默误判

评审时要优先防止“误识别为聊天页”这类假阳性，因为它会直接污染后续 Autofill 和生成链路。

### 4.5 Accessibility 与 OCR 降级关系

根据工程规格，提取优先级固定为 Accessibility -> OCR fallback。评审要检查：

- 是否仍以 Accessibility 作为首选路径
- Accessibility 失败时是否保留进入 OCR 的清晰条件
- 是否避免把 OCR 作为默认主路径
- 是否有超时、失败、权限缺失下的降级提示或返回

任何绕过这个优先级、或者把 OCR 直接写死为主流程的改动，都应视为设计偏离。

### 4.6 Autofill 安全性

自动填入是功能亮点，也是风险中心。评审至少确认：

- 首选 `ACTION_SET_TEXT`
- 填入后是否有回读验证或等价确认机制
- 失败时是否回退到剪贴板或明确提示
- 是否绝不调用发送动作、回车发送或其它可能造成自动发送的路径
- 是否只在确认目标输入框后执行填入

这个项目里，任何“为了方便直接帮用户发出去”的实现都应该直接拦截。

### 4.7 隐私与安全

评审时默认按“聊天内容是敏感数据”处理：

- 是否避免把完整聊天内容长期落盘
- 是否避免把截图、OCR 原文、API Key 打进日志
- OCR 调试图是否只在 Debug 构建限量保存，且 Release / 非调试构建不会落盘
- 网络请求中是否只发送必要上下文
- 是否对异常日志做最小必要记录
- 是否没有为了调试方便而引入危险的明文缓存

如果确实需要临时诊断信息，应该评估是否可只记录摘要、长度、阶段和错误码，而不是记录原文。

### 4.8 错误处理与可观测性

这个项目后续会接多种外部依赖，评审时不要只看 happy path：

- 失败是否能区分权限问题、页面识别失败、提取失败、网络失败、填入失败
- 是否保留足够的结构化日志帮助定位阶段性问题
- 错误提示是否能引导到下一步动作，而不是只抛通用失败
- 是否避免吞异常

没有日志也许现在还能开发，但没有阶段信息会让后续联调成本迅速失控。

### 4.9 测试与验证

当前项目还在骨架期，不要求一次性补齐完整自动化测试，但评审必须确认“怎么证明它没坏”：

- 是否至少说明了本次改动的验证方式
- 涉及纯逻辑、状态流转、解析规则时，是否有单元测试价值
- 涉及 Android 组件联动时，是否说明了手动验证场景
- 涉及微信识别、Autofill、权限链路时，是否写清真机验证前提

如果没有自动化测试，PR 描述里至少应写清：

- 验证设备或环境
- 触发步骤
- 预期结果
- 失败/降级结果

### 4.10 文档一致性

这个仓库文档约束很强，评审时要确认：

- 改动是否和 PRD、工程规格、当前状态文档一致
- 若实现刻意先落骨架，文档是否仍然准确表达“已实现”和“待实现”边界
- 若引入新的模块、接口或术语，是否需要补文档入口

在这个项目里，“代码改了但文档仍像旧世界”会直接误导后续 AI 和开发者。

## 5. 按改动类型的额外检查点

### 5.1 UI / Compose 改动

- 是否只是承担展示和交互，不吞掉业务边界
- 权限入口、提示文案是否与当前真实能力匹配
- 是否避免展示用户以为已经支持、但实际上尚未落地的能力

### 5.2 AccessibilityService 改动

- 是否控制触发频率和事件噪声
- 是否避免在 Service 中塞入过多状态和 UI 逻辑
- 是否考虑服务不可用、权限关闭、节点为空等场景

### 5.3 LLM 接入改动

- 是否有超时、取消、失败处理
- 是否避免把 Provider 细节扩散到上层
- 是否保证输出仍然服从“仅生成候选，不自动发送”

### 5.4 存储与日志改动

- 是否真的有必要落盘
- 字段是否最小化
- 是否定义清楚保留周期、清理策略或至少预留清理能力

## 6. 建议的评审评论模板

可按下面结构给出评审意见：

1. 结论：通过 / 有条件通过 / 需修改
2. 主要风险：指出最关键的 1-3 个问题
3. 边界判断：是否符合 MVP、分层和 Adapter 约束
4. 验证情况：本次是否有足够的验证说明
5. 后续建议：哪些问题可以留到后续 PR 解决

示例：

```md
结论：需修改

1. 当前把微信页面判断直接写进了 AccessibilityService，破坏了 Adapter 隔离，后续会让识别规则散落。
2. Autofill 失败后没有收口，状态可能停留在处理中，下一次触发会异常。
3. PR 没写真机验证步骤，当前这类改动至少要说明设备、页面、触发方式和失败兜底结果。

建议先把页面识别下沉到 WeChatAdapter，并补失败收口，再重新验证。
```

## 7. 合入前最低要求

对当前项目，合入前最低要求建议是：

- 目标清晰：这次改动属于当前阶段任务，而不是顺手扩范围
- 边界清晰：代码位置和职责基本合理
- 风险可控：失败、降级、隐私、自动填入安全性已考虑
- 可验证：至少有与改动匹配的验证说明

如果一个 PR 做不到这四点，宁可拆小，也不要硬合。
